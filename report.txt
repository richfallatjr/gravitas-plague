Gravitas Blender Stitcher Rotation Methodology
=============================================

Scope of this rollback
----------------------

The Blender stitcher was rolled back to the git-backed v1.7 tool behavior for
source rotation. The only code changes kept after the rollback are:

1. Head Transition and Tail Transition can be enabled or disabled independently.
2. Output frame math now uses only the enabled transition lengths.
3. Output frame math is asserted before any export happens.
4. The existing forced scene/action/USD export frame range path remains in use.

Working rotation methodology preserved from git
-----------------------------------------------

The working rotation path is the full source-pose world matrix solve. When
Rotate Head Source Pose or Rotate Tail Source Pose is enabled, the tool samples
each source pose bone and computes:

    target_world_matrix =
        source_rotation_matrix
        @ source_armature.matrix_world
        @ source_pose_bone.matrix

Then, when that sampled pose is applied to the current armature, it computes:

    target_armature_space =
        current_armature.matrix_world.inverted()
        @ target_world_matrix

    current_pose_bone.matrix = target_armature_space

This is the methodology that worked: rotate the full sampled source pose in
world space, then solve that pose into the current armature's local space.

The preserved git-backed v1.7 tool builds source_rotation_matrix from the
existing XYZ rotation fields and the existing pivot selector. No new rotation
behavior was added in this rollback.

What was intentionally not kept
-------------------------------

The rollback does not use the later experimental rotation approaches. It keeps
the git-backed full source-pose world matrix solve and only layers transition
toggles and frame math around it. It also does not add extra tail frames beyond
the configured Tail Transition Frames.

Head and tail behavior
----------------------

Head and tail now use the same methodology:

- if the transition is enabled, the source file is imported and sampled
- if source rotation is enabled, the same full world-pose solve is used
- if the transition is disabled, that source file is not required
- verification only requires keys for enabled transitions

Frame range rule
----------------

The output length is now checked exactly:

    body_frame_count = current_body_end - current_body_start + 1
    expected_output_length =
        body_frame_count
        + enabled_head_transition_frames
        + enabled_tail_transition_frames
        + enabled_tail_hold_frames

For a body range of 1-34 with head disabled, tail enabled, Tail Transition
Frames = 5, and Tail Hold Frames = 0:

    output_start = 1
    body_start_out = 1
    body_end_out = 34
    tail_target_out = 39
    output_end = 39

So the tail target is bodyEnd + 5, not bodyEnd + 30.
